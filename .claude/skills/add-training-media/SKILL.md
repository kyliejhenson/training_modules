# Add Training Media

This skill adds images to a training PowerPoint that was previously built by the generate-training skill. It exists as a separate step because media insertion into PPTX files requires careful coordination of XML relationships, media files, and slide elements — and doing this in a focused pass after the text content is finalized produces much more reliable results.

## Important: Read the PPTX Skill First

Before doing anything else, read the pptx SKILL.md file (check available skills for the path). You'll need its unpack/pack workflow and scripts. Also read the TEMPLATE_GUIDE.md in the workspace root for slide type reference.

---

## Step 1: Identify the Training Deck

Ask the user which training PPTX to add media to, if it's not obvious from context. The file will typically be in the workspace folder with a name like `Training_[Feature]_[Profile].pptx`.

Also identify the feature name (e.g., "skills", "mcp") since you'll need it to find the right images.

---

## Step 2: Analyze the Deck

Unpack the PPTX using the pptx skill's unpack script:

```bash
python scripts/office/unpack.py Training_Skills_Developers.pptx unpacked/
```

Then analyze it to understand which slides exist and what content they contain:

1. Run `markitdown` on the PPTX to get a text summary of all slides
2. Run `thumbnail.py` to get a visual grid
3. Read the slide XML files to identify:
   - Which slides are Full-Screen Image slides (they'll have placeholder shapes with the gray background and dashed circle from the template)
   - Which content slides could benefit from a supporting image
   - Which slides already have the small 256×256 template placeholder icons

The template's Full-Screen Image slides (originally slides 6 and 11) have a distinctive structure: a light gray (`E8E6DC`) background rectangle, a dashed orange circle, and an image icon. These placeholder shapes should be removed and replaced with the actual full-bleed image.

---

## Step 3: Match Images to Slides

Scan the `Imagery/` folder to find available images:

### Image sources (in priority order)

1. **Feature-specific screenshots**: `Imagery/images_<feature>/` (e.g., `images_skills/`, `images_mcp/`). These are 1920×1080 screenshots and are ideal for Full-Screen Image slides.

2. **Illustrations**: `Imagery/illustrations/` organized by theme:
   - `networks_and_nodes/` — network diagrams, connections
   - `hands_and_interaction/` — hands-on concepts
   - `data_and_analytics/` — data patterns
   - `objects_and_concepts/` — general concepts

3. **Hero illustrations**: `Imagery/illustrations/hero_illustrations/` — large, visually striking illustrations. **Only use these for hero slides** such as section dividers, title slides, or slides with very little visual content. Do NOT use hero illustrations on content-heavy slides where they would compete with text or diagrams.

4. **Logos**: `Imagery/logos/` for title/closing slides

### Matching strategy

Read each slide's text content and match it to the most relevant image based on what the slide is teaching. For example:
- A slide about "skill file structure" → `vscode_skill_md_frontmatter_name_description_allowed_tools_model.png`
- A slide about "discovering skills" → `claude_code_discover_plugins_list_with_install_counts.png`
- A slide about "running skills" → `claude_code_terminal_pr_review_skill_loading.png`

Don't force every slide to have an image. Focus on:
- **Full-Screen Image slides**: These MUST get a full-bleed feature screenshot
- **Content slides** with concepts that are clearer with a visual: Add a supporting image
- **Title/Closing slides**: Consider adding the Anthropic logo

### Placeholder icon replacement

When a slide already has a placeholder icon (the small 256×256 template icons), follow this process to choose the best replacement:

1. **Review the slide context**: Read the slide's title, body text, and presenter notes to understand the concept being taught.
2. **Match by icon title**: Scan the available icons in the imagery folders and compare their filenames/titles against the slide's topic. Choose the icon whose title is most semantically relevant to the slide content — not just a generic match, but the closest conceptual fit.
3. **Choose the right color variant**: Icons often come in multiple color variants (e.g., dark on light, light on dark, orange accent, neutral). Inspect the slide's background color and surrounding element colors, then pick the variant that:
   - Has sufficient contrast against the slide background
   - Complements the slide's color scheme (e.g., use a light/white variant on dark backgrounds, a dark variant on light backgrounds)
   - Matches the overall visual tone of the slide (e.g., orange accent icons work well on slides that already use the brand orange)

Present the proposed image-to-slide mapping to the user and ask for confirmation before proceeding.

## Step 4: Build the Media Manifest

Create a JSON manifest that maps slide filenames to media operations. The manifest tells the `insert_media.py` script exactly what to do.

### Manifest format

```json
{
    "slide6.xml": {
        "images": [
            {
                "src": "/absolute/path/to/screenshot.png",
                "position": "full-bleed",
                "caption": "Skill file structure in VS Code",
                "source": "VS Code with Claude extension"
            }
        ]
    },
    "slide12.xml": {
        "images": [
            {
                "src": "/absolute/path/to/icon.png",
                "position": "replace",
                "replace_index": 0
            }
        ]
    }
}
```

### Image position types

- **`full-bleed`**: Image fills the entire slide (10" × 5.625"). Use for Full-Screen Image slides. The script inserts the image as the first shape in the tree so it sits behind the caption bar.

- **`replace`**: Swaps an existing placeholder image file on disk. Use this for the small 256×256 template icons on content slides — it keeps the existing positioning but replaces the generic icon with a relevant image. Set `replace_index` to target a specific image if the slide has multiple (0-indexed).

- **`content`**: Adds a new image at specific coordinates. Provide `x`, `y`, `cx`, `cy` in EMU (914,400 EMU = 1 inch). Use when you need precise control over placement.

Save the manifest as `media_manifest.json` in the working directory.

---

## Step 5: Prepare Full-Screen Image Slides

Before running the insertion script, you need to clean up Full-Screen Image slides by removing the template's visual placeholders. These are the shapes that make up the "placeholder" look — the gray background rectangle, the dashed circle, and the image icon.

For each Full-Screen Image slide in the manifest:

1. Read the slide XML
2. Identify and remove the placeholder shapes. They're `<p:sp>` elements — look for:
   - A rectangle with gray fill (`E8E6DC`) that covers most of the slide
   - A circle/ellipse shape with dashed outline (the "drop image here" indicator)
   - Any small icon shapes within the placeholder area
3. Keep the caption bar at the bottom (the semi-transparent dark rectangle with `[Image Caption]` and `[Source]` text) — this should stay on top of the full-bleed image
4. Update the caption text with the `caption` and `source` values from the manifest

Use the Edit tool for these changes, not sed or scripts. The Edit tool's specificity prevents accidental corruption of the XML.

---

## Step 6: Insert Media

Run the media insertion script:

```bash
python <skill-path>/scripts/insert_media.py <unpacked_dir> media_manifest.json
```

The script handles:

- Copying image files to `ppt/media/` with proper naming
- Adding `<Relationship>` entries to each slide's `.rels` file
- Building and inserting `<p:pic>` XML elements with correct coordinates
- Ensuring `[Content_Types].xml` has entries for all image formats used
- For `replace` operations: swapping the media file on disk while preserving XML references
- For `full-bleed` operations: inserting the image as the first shape (behind text overlays)

Check the script output for any errors. Common issues:
- `Image not found`: Check the source path in the manifest
- `rels not found`: The slide filename in the manifest doesn't match the actual file

---

## Step 7: Clean and Pack

After insertion, clean and repack:

```bash
python scripts/clean.py <unpacked_dir>
python scripts/office/pack.py <unpacked_dir> <output.pptx> --original <original.pptx>
```

Use the same output filename as the input (overwrite it) unless the user wants to keep the text-only version.

---

## Step 8: Visual QA

This step is important — media insertion is the operation most likely to produce visual artifacts.

Convert the output to images and inspect every slide that received media:

```bash
soffice --headless --convert-to pdf <output.pptx> --outdir /tmp/
pdftoppm -png -r 150 /tmp/<output>.pdf /tmp/slide
```

Check for:
- **Full-bleed images**: Do they fill the slide? Is the caption bar visible on top?
- **Content images**: Are they properly positioned? Not overlapping text?
- **Replaced icons**: Did the new image render at the right size? Does the color variant look correct against the slide background?
- **Hero illustrations**: Are they only on section dividers or low-content slides — not on content-heavy slides?
- **No broken references**: All slides should render cleanly with no error icons

If anything looks wrong, read the specific slide XML and fix it with the Edit tool, then repack and check again.

## EMU Quick Reference

When specifying custom positions, these conversions are helpful:

| Measurement | EMU Value |
|------------|-----------|
| 1 inch | 914,400 |
| Full slide width (10") | 9,144,000 |
| Full slide height (5.625") | 5,143,500 |
| 0.5 inch margin | 457,200 |
| 0.7 inch margin | 640,080 |
| Half slide width | 4,572,000 |

Common content image placements:
- **Right-aligned, half-width**: x=4572000, y=914400, cx=4114800, cy=3429000
- **Centered, medium**: x=2286000, y=1143000, cx=4572000, cy=2571750
- **Small icon**: x=640080, y=274320, cx=502920, cy=502920
